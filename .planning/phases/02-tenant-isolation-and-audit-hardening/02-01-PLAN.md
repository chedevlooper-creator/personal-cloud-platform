# Plan 02-01: Audit And Enforce DB Resource Scoping

## Objective

Make representative DB-backed reads, writes, and deletes enforce tenant
ownership at the persistence boundary, not only through earlier route or service
call assumptions.

## Target Files

- `services/workspace/src/service.ts`
- `services/workspace/src/datasets/service.ts`
- `services/runtime/src/service.ts`
- `services/publish/src/service.ts`
- `services/agent/src/channels/router.ts`
- Existing or new focused tests beside the touched services.

## Must Haves

- Touched sensitive mutations include `userId`, scoped workspace ownership, or
  an equivalent tenant predicate in the final DB operation.
- Cross-tenant negative tests cover at least one high-risk operation per touched
  service area.
- Route contracts remain stable unless an existing route is unsafe.
- No broad repository extraction or schema rewrite.
- No new `any` or broad `z.any()` shortcuts.

## Threat Model

| Threat                                           | Mitigation In This Plan                                                     |
| ------------------------------------------------ | --------------------------------------------------------------------------- |
| Caller passes another tenant's resource id       | DB operation includes ownership predicate or rejects through scoped lookup. |
| Future refactor removes pre-read guard           | Tests fail because final operation no longer enforces tenant boundary.      |
| Channel/external id links task from another user | Task polling resolves task ownership before returning completion state.     |

## Tasks

1. Inventory ID-only DB reads/mutations in the target files and classify which
   ones are protected only by prior scoped reads.
2. Add failing tests for representative unsafe patterns:
   - dataset soft delete cannot affect another user's dataset.
   - runtime lifecycle update/delete cannot affect another user's runtime.
   - hosted service lifecycle update cannot affect another user's service.
   - channel task polling cannot observe another user's task.
3. Update final DB operations to include tenant predicates or scoped ownership
   joins/lookups.
4. Keep error responses behavior-compatible with current callers.
5. Run targeted service tests, then the repo test/typecheck gates if the change
   is not isolated.

## Verification Commands

```powershell
pnpm --filter @pcp/workspace-service test
pnpm --filter @pcp/runtime-service test
pnpm --filter @pcp/publish-service test
pnpm --filter @pcp/agent-service test
pnpm typecheck
```

## Done When

- Tests prove cross-tenant resource ids cannot affect or observe another user's
  resources in the touched flows.
- Tenant predicates are present at the DB side-effect boundary.
- No unrelated dirty worktree changes are staged.
