# Plan 02-02: Harden Storage Paths Runtime Labels And Audit Coverage

## Objective

Make tenant identity explicit in storage paths, runtime/publish container
metadata, and privileged-action audit events.

## Target Files

- `services/workspace/src/service.ts`
- `services/workspace/src/routes/snapshots.ts`
- `services/runtime/src/service.ts`
- `services/runtime/src/provider/docker.ts`
- `services/publish/src/service.ts`
- Existing or new focused tests beside the touched services.

## Must Haves

- Snapshot artifact keys include `userId` and `workspaceId`.
- Existing snapshot rows remain readable where practical.
- Runtime host workspace paths are tenant-prefixed and path traversal safe.
- Publish workspace volume paths are tenant-aware or resolved through a
  tenant-aware runtime/storage contract.
- Runtime and publish containers include safe `pcp.*` labels for tenant/resource
  correlation.
- Audit details are allow-listed and do not contain secrets, cookies, raw env
  vars, file contents, or provider credentials.

## Threat Model

| Threat                                                  | Mitigation In This Plan                                                |
| ------------------------------------------------------- | ---------------------------------------------------------------------- |
| Object key collision across tenants                     | Prefix snapshot/storage keys with user and workspace identifiers.      |
| Host path escape or workspace collision                 | Construct paths from normalized tenant segments under configured root. |
| Container cannot be attributed during incident response | Add non-sensitive `pcp.*` labels to runtime and publish containers.    |
| Audit log leaks sensitive payloads                      | Use small allow-listed detail objects and safe failure handling.       |

## Tasks

1. Add failing tests for snapshot key shape and runtime host path shape.
2. Introduce or update helpers so snapshot keys are shaped like
   `snapshots/{userId}/{workspaceId}/...`.
3. Preserve read compatibility for existing snapshot keys if the DB can contain
   old rows.
4. Update runtime host path construction to include both `userId` and
   `workspaceId`, normalized under `WORKSPACE_HOST_ROOT`.
5. Add runtime Docker labels:
   - `pcp.service`
   - `pcp.userId`
   - `pcp.workspaceId`
   - `pcp.runtimeId`
6. Add publish Docker labels:
   - `pcp.service`
   - `pcp.userId`
   - `pcp.workspaceId`
   - `pcp.hostedServiceId`
7. Normalize audit failure handling in touched snapshot/runtime/publish flows.
8. Add or update tests for privileged audit event detail shape.

## Verification Commands

```powershell
pnpm --filter @pcp/workspace-service test
pnpm --filter @pcp/runtime-service test
pnpm --filter @pcp/publish-service test
pnpm typecheck
```

## Done When

- Storage and host paths visibly include tenant identity.
- Docker config generation includes tenant/resource labels without leaking
  secrets.
- Audit details are structured and safe in touched privileged flows.
