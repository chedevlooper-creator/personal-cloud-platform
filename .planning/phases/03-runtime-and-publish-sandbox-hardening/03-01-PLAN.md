# Plan 03-01: Harden Docker Provider Defaults For Runtime And Publish

## Objective

Make runtime and publish Docker launch configs stricter and less ambiguous,
without requiring Docker in tests.

## Target Files

- `services/runtime/src/provider/docker.ts`
- `services/runtime/src/provider/docker.test.ts`
- `services/publish/src/service.ts`
- `services/publish/src/service.test.ts`

## Must Haves

- Runtime and publish container configs explicitly set:
  - `Privileged: false`
  - `Init: true`
  - `OomKillDisable: false`
  - `MemorySwap` equal to `Memory`
  - existing non-root, readonly rootfs, dropped caps, pids, tmpfs, ulimit, and
    `no-new-privileges` remain intact.
- Runtime containers keep `NetworkMode: none`.
- Publish containers keep `NetworkMode: pcp_network`.
- Hosted workspace bind stays read-only; runtime workspace bind remains writable.
- Tests fail if any new sandbox field is removed.

## Tasks

1. Add failing runtime provider test assertions for explicit Docker hardening
   fields.
2. Add failing publish service test assertions for the same hardening fields.
3. Implement minimal HostConfig changes in runtime provider.
4. Implement minimal HostConfig changes in publish service.
5. Run targeted runtime and publish tests.
6. Run root test/typecheck and record any unrelated known lint debt.

## Verification Commands

```powershell
pnpm --filter @pcp/runtime-service test
pnpm --filter @pcp/publish-service test
pnpm test
pnpm typecheck
```

## Done When

- Runtime and publish tests prove the hardened Docker defaults are present.
- No Docker daemon is required for verification.
- Phase 3 state and summary artifacts are updated.
