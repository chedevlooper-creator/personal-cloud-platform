# Plan 01-02: Standardize API Errors And Shared DTO Contracts

<objective>
Give services and the frontend one predictable error contract and reduce route
typing bypasses around cross-boundary DTOs.
</objective>

<context>
Read before editing:
- `.planning/phases/01-contracts-config-and-auth-foundation/01-CONTEXT.md`
- `.planning/phases/01-contracts-config-and-auth-foundation/01-RESEARCH.md`
- `packages/shared/README.md`
- `.planning/codebase/CONVENTIONS.md`
- Representative service route files and frontend API clients.
</context>

<must_haves>
## Must Haves

### Truths
- Cross-service/frontend schemas belong in `@pcp/shared`.
- Do not add broad `any`, `as any`, or `z.any()` shortcuts.
- Client-facing errors must not leak internal stack traces or raw internal
  exception messages.
- Frontend callers must be able to parse errors consistently.

### Phase Success Criteria Covered
- API clients receive a consistent error envelope without internal stack leaks.
- Route schemas use shared DTOs where contracts cross service/frontend
  boundaries.
</must_haves>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Internal errors leak implementation details | High | Central Fastify error handler maps to safe envelope. |
| Invalid request payload reaches service logic | Medium | Zod schemas at route boundary, shared DTOs for cross-boundary contracts. |
| Frontend mishandles failures due to mixed shapes | Medium | One envelope and small client parsing helper. |
| Type bypass hides route contract regressions | Medium | Remove or avoid new `as any`/`z.any()` in touched routes. |
</threat_model>

<tasks>
## Tasks

<task id="1" type="analysis">
Inventory existing route error shapes and `as any`/`z.any()` hotspots in touched
services. Pick a focused initial migration set that covers representative
patterns without rewriting every route at once.
</task>

<task id="2" type="implementation">
Define a small shared API error envelope and error code/type model. Put
cross-boundary schema/types in `@pcp/shared` while keeping the package
source-only.
</task>

<task id="3" type="implementation">
Add service-level Fastify error mapping that translates known domain errors,
validation failures, auth failures, and unknown errors into the shared envelope.
</task>

<task id="4" type="implementation">
Update representative route handlers to throw or return through the new error
path and to use shared DTO schemas where appropriate.
</task>

<task id="5" type="implementation">
Update frontend API helpers/hooks only as needed so user-facing surfaces can
read the new envelope without breaking current behavior.
</task>

<task id="6" type="test">
Add route tests with `fastify.inject()` for validation failure, auth failure,
known domain error, and unknown error mapping in representative services.
</task>
</tasks>
